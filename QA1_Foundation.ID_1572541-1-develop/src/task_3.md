| Методология | Плюсы | Минусы / Риски | Пример / Тип проекта |
|:---|:---|:---|:---|
| **Waterfall** | - Простота управления: четкие этапы <br>- Легкость оценки бюджета | - Невозможно вернуться назад <br>- Заказчик видит продукт в конце | **Гос. системы / Медицина** <br> *Пример:* Система учёта для налоговой |
| **V-Model** | - Высокое качество за счёт параллельного тестирования <br>- Снижение рисков на ранних этапах | - Сложно при изменении требований <br>- Требует много тестовой документации до кодирования | **Авиация / Автопром** <br> *Пример:* ПО для тормозной системы авто |
| **Incremental** | - Быстрый выход первой версии (MVP) <br>- Частичная окупаемость на ранних этапах | - Сложность интеграции модулей <br>- Риск неверного финального продукта | **Крупные ERP-системы** <br> *Пример:* Внедрение системы в банке по модулям |
| **Scrum** | - Предсказуемая поставка каждые 2 недели <br>- Высокая адаптивность к изменениям | - Требует дисциплины и опыта <br>- Оверхед на церемониях (15% времени) | **Стартапы / Продукты** <br> *Пример:* Мобильное приложение для доставки еды |
| **Kanban** | - Минимальный порог входа <br>- Сокращение времени за счёт лимитов WIP | - Нет дедлайнов <br>- Сложность с прогнозом даты релиза | **Служба поддержки** <br> *Пример:* IT-поддержка банка с заявками |




| Идея манифеста | Что это значит на практике? |
|:---|:---|
| **Люди и взаимодействие важнее процессов и инструментов** | 1. Ежедневная летучка с вопросами «Что сделал вчера? Что сделаю сегодня? Какие блокеры?». <br> 2. Раз в 2 недели обсуждаем не код, а процесс («Что пошло хорошо/плохо? Что улучшим?»). |
| **Работающий продукт важнее исчерпывающей документации** | 1. Простейшая версия функции за 2 дня и показываем заказчику. <br> 2. Чистый код + тесты (они выполняют роль живой документации). |
| **Сотрудничество с заказчиком важнее согласования условий контракта** | 1. Приглашаем заказчика в конце спринта — он сам крутит новую функцию и даёт обратную связь сразу. <br> 2.  Представитель бизнеса сидит с командой в одном пространстве, разработчик подходит и уточняет нюансы за минуту. |
| **Готовность к изменениям важнее следования первоначальному плану** | 1. Пересортировываем задачи каждую неделю. <br> 2.  План только на ближайшие 2 недели, даже если появилась новая функция — легко перепланировать. |



**«То есть не отрицая важности того, что справа, мы всё-таки больше ценим то, что слева»**

По расположению текста в Agile-манифесте можно понять что не отрицая процессов, документации, контрактов, плану важнее взаимодействовать с командой, заказчиком, сотрудничать и быть готовым к изменениям
